## 🎯 نبذة عن المشروع

نظام إدارة المتاجر متعددة الفروع هو منصة مؤسسية متكاملة مصممة لإدارة شبكات البيع بالتجزئة المعقدة. يوفر النظام حلولاً شاملة لإدارة الفروع، تعيين الموظفين، تتبع المخزون، وإدارة المنتجات مع واجهة مستخدم سلسة وتجربة مستخدم محسّنة.

### 🌟 النقاط البارزة

- **إدارة الفروع الشاملة**: إنشاء، تعديل، وحذف الفروع مع إدارة كاملة للمعلومات التفصيلية
- **نظام الصلاحيات المتقدم**: إدارة أدوار متعددة (مسؤول، مدير متجر، موظف) مع صلاحيات دقيقة
- **التعيين الذكي للموظفين**: نظام متطور لتعيين المديرين والموظفين للفروع مع حل التعارضات
- **إدارة المنتجات المرتبطة**: ربط المنتجات بالفروع مع تتبع التوافر
- **نظام البروشور الرقمي**: رفع وعرض وتحميل بروشورات الفروع بصيغة PDF
- **البحث الذكي**: محرك بحث متقدم يدعم البحث عن الفروع، المنتجات، والموظفين
- **لوحة معلومات تفاعلية**: إحصائيات حية ومؤشرات أداء للفروع والموظفين

## ✨ الميزات الأساسية

### 🔐 نظام المصادقة والصلاحيات

- **تسجيل دخول آمن**: نظام مصادقة متعدد الطبقات مع دعم Laravel Sanctum
- **إدارة الأدوار**: نظام صلاحيات مرن يعتمد على Spatie Laravel Permission
- **صلاحيات دقيقة**: تحكم دقيق في الوصول لكل جزء من النظام
- **حماية المسارات**: Middleware متقدم لحماية المسارات بناءً على الأدوار

### 🏢 إدارة الفروع

- **إدارة شاملة**: CRUD كامل للفروع مع التحقق من صحة البيانات
- **المعلومات التفصيلية**: إدارة كاملة لمعلومات الاتصال، الموقع، وساعات العمل
- **البروشور الرقمي**: رفع وعرض بروشورات PDF مع دعم التحميل المتجزئ (Chunked Upload)
- **التعيينات**: تعيين المديرين والموظفين للفروع مع حل التعارضات تلقائيًا
- **المنتجات المرتبطة**: ربط المنتجات بالفروع مع تتبع التوافر

### 👥 إدارة الموظفين

- **تعيين الموظفين**: نظام متطور لتعيين الموظفين للفروع
- **إدارة المديرين**: تعيين مدير واحد لكل فرع مع حل التعارضات
- **تتبع الأداء**: إحصائيات مفصلة عن أداء الموظفين والفروع
- **سجلات التغيير**: تتبع كامل لجميع التغييرات على التعيينات

### 📦 إدارة المنتجات

- **إدارة المنتجات**: CRUD كامل للمنتجات مع التحقق من صحة البيانات
- **الربط بالفروع**: ربط المنتجات بالفروع مع تتبع التوافر
- **البحث المتقدم**: بحث ذكي عن المنتجات مع فلاتر متعددة
- **عرض تفصيلي**: صفحات تفاصيل شاملة للمنتجات مع الفروع المرتبطة

### 🔍 البحث الذكي

- **بحث متعدد الأبعاد**: بحث عن الفروع، المنتجات، والموظفين في واجهة واحدة
- **فلاتر متقدمة**: فلاتر متعددة لتضييق نتائج البحث
- **نتائج فورية**: بحث AJAX للحصول على نتائج فورية
- **إجراءات سريعة**: إجراءات مباشرة من نتائج البحث

## 🏗️ البنية المعمارية

### 🎨 البنية الموجهة للخدمات (SOA)

يطبق النظام نمط **طبقة الخدمات** بدقة، مما يفصل منطق الأعمال تمامًا عن مخاوف HTTP. يتم الاحتفاظ بـ Controllers تحت 50 سطر، وتعمل كطبقات تنسيق فقط تفوض إلى خدمات متخصصة:

- **`StoreCrudService`** — يدير عمليات CRUD الذرية المغلفة في معاملات قاعدة البيانات
- **`StoreSearchService`** — يغلف منطق البحث المعقد مع تكامل الإجراءات المخزنة
- **`StoreBrochureService`** — يدير عرض PDF، التخزين المؤقت، والتحميلات المتجزئة
- **`AssignmentService`** — ينسق مزامنة الموظفين بالفروع مع حل التعارضات
- **`StoreStatsService`** — يحسب تحليلات لوحة المعلومات مع استراتيجيات التخزين المؤقت

هذه البنية تمكن **التوسع الأفقي** لمنطق الأعمال، **الاختبار الوحدوي** في عزلة، و **إعادة استخدام الكود** عبر واجهات مختلفة (HTTP، CLI، API).

### 🔒 سلامة الأنواع الصارمة

كل ملف PHP يعلن `strict_types=1`، مما يفرز فحص الأنواع في وقت الترجمة. تتضمن النماذج خصائص **PHPDoc شاملة** للإكمال التلقائي في IDE، وتتميز جميع الطرق ب **تلميحات أنواع كاملة**:

```php
declare(strict_types=1);

class Store extends Model
{
    /** @property int $id */
    /** @property string $name */
    /** @property-read Province $province */
    
    public function scopeFilterName(Builder $query, ?string $name): Builder
    {
        // بناء استعلام آمن من حيث الأنواع
    }
}
```

هذا يزيل فئات كاملة من أخطاء وقت التشغيل ويمكن **تحليل PHPStan المستوى 5** الثابت لالتقاط الأخطاء قبل التنفيذ.

## 🎭 أنماط التصميم والمبادئ

### ⚖️ مبادئ SOLID في التطبيق

**المسؤولية الفردية** — كل خدمة تتعامل مع قلق مجال واحد. `StoreCrudService` يدير فقط الاستمرارية، بينما `StoreBrochureService` يتعامل فقط مع عمليات الملفات.

**المفتوح/المغلق** — تتم إضافة الميزات الجديدة من خلال **مستمعي الأحداث** و **البرمجيات الوسيطة** دون تعديل الخدمات الموجودة. مستمع `InvalidateStoreCache` يتعامل مع مسح التخزين المؤقت لأي تعديل على المتجر تلقائيًا.

**عكس التبعية** — تعتمد جميع الخدمات على التجريدات (واجهات) يتم حقنها عبر حاوية Laravel. تتلقى Controllers الخدمات عبر حقن المُنشئ، مما يسهل المحاكاة والاختبار.

### 📦 نمط المستودع

تعمل طبقة الخدمات ك **تجريد مستودع**، مما يخفي تفاصيل تنفيذ Eloquent من Controllers. هذا يسمح بـ:

- **تبديل مصادر البيانات** (MySQL → PostgreSQL) دون تغييرات في Controllers
- **التخزين المؤقت على مستوى المستودع** بشفافية
- **تكوين الاستعلام المعقد** في الخدمات، مما يبقي Controllers خفيفة

### 🎯 نمط الاستراتيجية للتفويض

تطبق `StoreScopeService` **نمط الاستراتيجية** لنطاق البيانات بناءً على الأدوار:

```php
interface StoreScopeStrategy
{
    public function applyScope(Builder $query, User $user): Builder;
}

class AdminScopeStrategy implements StoreScopeStrategy
{
    public function applyScope(Builder $query, User $user): Builder
    {
        return $query; // لا قيود للمسؤولين
    }
}

class ManagerScopeStrategy implements StoreScopeStrategy
{
    public function applyScope(Builder $query, User $user): Builder
    {
        return $query->where('manager_id', $user->id);
    }
}
```

هذا يجعل إضافة أدوار جديدة أو تعديل قواعد الوصول **تغييرات غير مكسورة**.

## 🚀 الميزات المتقدمة

### ⚡ إبطال التخزين المؤقت القائم على الأحداث

ينفذ النظام **إبطال التخزين المؤقت التلقائي** عبر نظام أحداث Laravel:

```php
Event::listen(StoreCreated::class, InvalidateStoreCache::class);
Event::listen(StoreUpdated::class, InvalidateStoreCache::class);
Event::listen(StoreDeleted::class, InvalidateStoreCache::class);
```

عند حدوث أي تعديل على المتجر، يقوم مستمع `InvalidateStoreCache` تلقائيًا بمسح التخزين المؤقت ذي الصلة. هذا **يزيل أخطاء توقف التخزين المؤقت** ويزيل مسح التخزين المؤقت اليدوي من منطق الأعمال.

### 🔄 المعالجة غير المتزامنة القائمة على الطابور

يتم تفويض العمليات الثقيلة مثل تحميلات البروشور إلى **وظائف الطابور**:

```php
class ProcessBrochureUpload implements ShouldQueue
{
    public int $tries = 3;
    public int $timeout = 300;
    
    public function handle(StoreBrochureService $service): void
    {
        // معالجة PDF بشكل غير متزامن
    }
}
```

هذا يوفر:
- **تجربة مستخدم غير محجوبة** — التحميلات لا تشغل عمال HTTP
- **إعادة المحاولة التلقائية** عند الفشل مع تراجع أسي
- **التوسع الأفقي** — عدة عمال طابور يعالجون الوظائف بالتوازي
- **طوابير الرسائل الميتة** لتحليل الوظائف الفاشلة

### 📤 التحميلات المتجزئة

يتم تحميل بروشورات PDF الكبيرة في **أجزاء** لتجاوز حدود حجم الخادم:

- **إمكانية الاستئناف** — يمكن استئناف التحميلات المقاطعة
- **كفاءة الذاكرة** — يتم معالجة الأجزاء بشكل فردي
- **تتبع التقدم** — تقدم التحميل الفوري للمستخدمين
- **التحميلات المتزامنة** — يمكن تحميل عدة أجزاء بالتوازي

### 🗄️ تكامل الإجراءات المخزنة

يتم تفويض عمليات البحث المعقدة في المتاجر إلى **إجراءات MySQL المخزنة** للأداء:

```sql
CREATE PROCEDURE SearchStores(
    IN p_name VARCHAR(255),
    IN p_province_id INT,
    IN p_status VARCHAR(50),
    IN p_phone VARCHAR(20)
)
```

هذا يقلل **مشاكل الاستعلام N+1** ويستفيد من تحسينات جانب قاعدة البيانات للصلات والتصفية المعقدة.

## ⚡ تحسينات الأداء

### 🗄️ تحسينات على مستوى قاعدة البيانات

- **التحميل المسبق** — يتم تحميل العلاقات مسبقًا باستخدام `with()` لمنع استعلامات N+1
- **نطاقات الاستعلام** — أجزاء الاستعلام القابلة لإعادة الاستخدام تؤلف الفلاتر المعقدة بكفاءة
- **استراتيجية الفهرسة** — فهارس مركبة على الأعمدة المطلوبة بشكل متكرر
- **الإجراءات المخزنة** — عمليات البحث المعقدة تنفذ على خادم قاعدة البيانات
- **تجميع الاتصالات** — إدارة اتصال Laravel تقلل من الحمل

### 💾 استراتيجية التخزين المؤقت

ينفذ التطبيق **استراتيجية تخزين مؤقت متعددة الطبقات**:

```php
// تخزين قوائم المتاجر لمدة ساعة
Cache::remember('stores.active', 3600, fn () => 
    Store::active()->with(['province', 'manager'])->get()
);

// تخزين المتاجر الفردية مع العلامات
Cache::tags(['stores'])->remember("store.{$id}", 7200, fn () =>
    Store::with(['employees', 'products'])->findOrFail($id)
);
```

- **الإبطال القائم على العلامات** — مسح مجموعات التخزين المؤقت بالكامل بشكل ذري
- **الانتهاء التلقائي** — وقت البقاء يمنع البيانات القديمة
- **تسخين التخزين المؤقت** — التعبئة المسبقة للتخزين المؤقت أثناء النشر

- **Vite** — HMR سريع وبنيات إنتاج محسّنة

## 🔒 بنية الأمان

### 🛡️ الدفاع في العمق

ينفذ التطبيق **طبقات أمان متعددة**:

1. **تحديد المعدل** — حدود قابلة للتكوين لكل نقطة نهاية تمنع الإساءة:
   ```php
   RateLimiter::for('login', fn (Request $r) => 
       Limit::perMinute(5)->by($r->ip() . '|' . $r->input('email'))
   );
   ```

2. **التحقق من صحة الإدخال** — طلبات النماذج تعقم وتتحقق قبل الوصول إلى الخدمات
3. **بوابات التفويض** — السياسات تتحقق من الأذونات قبل الوصول إلى البيانات
4. **منع حقن SQL** — ربط معاملات Eloquent افتراضيًا
5. **حماية XSS** — Blade يهرب من المخرجات تلقائيًا؛ رؤوس CSP مكونة
6. **حماية CSRF** — رموز على جميع الطلبات التي تغير الحالة
7. **مسارات التدقيق** — `created_by` و `updated_by` تتبع جميع التعديلات

### 👥 التحكم في الوصول بناءً على الأدوار (RBAC)

باستخدام **Spatie Laravel Permission**، يطبق النظام:

- **أدوار هرمية** — الأدوار يمكن أن ترث الأذونات من الأدوار الأصل
- **تخزين الأذونات مؤقتًا** — عمليات التحقق من الأذونات مخزنة مؤقتًا للأداء
- **أذونات ديناميكية** — تعيين الأذونات في وقت التشغيل دون تغييرات في الكود
- **تكامل البرمجيات الوسيطة** — فرض الأذونات على مستوى المسار

### 🔐 معالجة الملفات الآمنة

- **تجريد التخزين** — الملفات مخزنة خارج جذر الويب
- **عناوين URL الموقعة** — روابط تنزيل مؤقتة منتهية الصلاحية
- **التحقق من MIME** — التحقق من نوع الملف عند التحميل
- **التحميلات المتجزئة** — معالجة فعالة من حيث الذاكرة

## 🎯 سلامة الأنواع وجودة الكود

### 🔍 التحليل الثابت مع PHPStan

تكوين PHPStan المستوى المستوى 5 يفرز:

- **قواعد المقارنة الصارمة** — يمنع أخطاء إجبار الأنواع
- **كشف الإهمال** — يحدد استخدام الكود القديم
- **كشف الكود غير المستخدم** — يزيل الكود الميت
- **فحص الأنواع العامة** — يضمن سلامة أنواع المجموعات

```bash
composer stan  # تشغيل تحليل PHPStan
```

### 🎨 تنسيق الكود الآلي

يضمن Laravel Pint **نمط كود متسق** عبر المشروع:

```bash
composer pint      # فحص نمط الكود
composer pint:fix  # إصلاح مشاكل النمط تلقائيًا
```

يتبع جميع الكود **معايير PSR-12** تلقائيًا.

### 🧪 الاختبار الحديث مع PHPUnit

يوفر PHPUnit **اختبارات معبرة وقابلة للقراءة**:

```php
test('admin can create store', function () {
    $admin = User::factory()->admin()->create();
    
    actingAs($admin)
        ->post(route('stores.store'), Store::factory()->raw())
        ->assertRedirect(route('stores.index'));
        
    expect(Store::count())->toBe(1);
});
```

- **معاملات قاعدة البيانات** — يتم التراجع عن الاختبارات تلقائيًا
- **المصانع** — توليد بيانات اختبار متسق
- **التنفيذ المتوازي** — تشغيل الاختبارات بالتوازي للسرعة

## 📡 البنية القائمة على الأحداث

### 🎉 أحداث المجال

يصدر النظام **أحداث المجال** للتغييرات الهامة في الحالة:

```php
class StoreCreated
{
    public function __construct(
        public readonly Store $store,
        public readonly ?int $actorId = null
    ) {}
}
```

الأحداث هي **حاملات بيانات غير قابلة للتغيير** تحتوي على جميع السياق ذي الصلة.

### 👂 مستمعي الأحداث

يستجيب المستمعون للأحداث **بشكل غير متزامن**:

```php
class InvalidateStoreCache
{
    public function handle(StoreCreated|StoreUpdated|StoreDeleted $event): void
    {
        $this->storeService->flushStoreCaches();
    }
}
```

هذا يتيح:
- **الاقتران السائب** — مصدرو الأحداث لا يعرفون عن المستمعين
- **قابلية التوسع** — إضافة مستمعين جدد دون تعديل الكود الموجود
- **تسجيل التدقيق** — يمكن تسجيل جميع تغييرات الحالة مركزيًا
- **الإشعارات** — إشعارات البريد الإلكتروني/SMS يتم تشغيلها بواسطة الأحداث

### 📢 بث الأحداث

يمكن بث الأحداث **إلى الواجهة الأمامية** عبر Laravel Echo للتحديثات الفورية:

```php
class StoreUpdated implements ShouldBroadcast
{
    public function broadcastOn()
    {
        return new Channel('stores');
    }
}
```

## 📈 اعتبارات التوسع

### 🔄 التوسع الأفقي

تدعم البنية **التوسع الأفقي**:

- **عمال HTTP عديمي الحالة** — لا يوجد تخزين جلسة في ذاكرة PHP
- **المعالجة القائمة على الطابور** — توزيع الوظائف الخلفية عبر العمال
- **طبقة التخزين المؤقت** — Redis مشترك عبر جميع الحالات
- **تجميع اتصال قاعدة البيانات** — إعادة استخدام الاتصال بكفاءة

### 🗄️ قابلية توسع قاعدة البيانات

- **النسخ المتماثلة للقراءة** — يمكن توجيه استعلامات القراءة إلى خوادم النسخ المتماثلة
- **جاهزية التجزئة** — يمكن تجزئة بيانات المتجر حسب المنطقة
- **تحسين الاستعلام** — الإجراءات المخزنة تقلل من حمل قاعدة البيانات
- **استراتيجية الترحيل** — دعم الترحيل بدون توقف

### 💾 استراتيجية التخزين المؤقت

- **التخزين المؤقت متعدد الطبقات** — تخزين مؤقت للتطبيق + CDN + تخزين مؤقت للمتصفح
- **تسخين التخزين المؤقت** — التعبئة المسبقة للتخزين المؤقت أثناء النشر
- **منع طوفان التخزين المؤقت** — آليات القفل تمنع القطعان
- **الإبطال الدقيق للتخزين المؤقت** — الإبطال القائم على العلامات للتحكم الدقيق

## 🚀 التثبيت والإعداد

### المتطلبات الأساسية

- PHP >= 8.2
- Composer
- MySQL >= 8.0
- Node.js >= 18
- Redis (اختياري للتخزين المؤقت والطوابور)

### خطوات التثبيت

1. **استنساخ المستودع**
   ```bash
   git clone https://github.com/Rami-Saab/Store-Management.git
   cd Store-Management
   ```

2. **تثبيت التبعيات**
   ```bash
   composer install
   npm install
   ```

3. **إعداد البيئة**
   ```bash
   cp .env.example .env
   php artisan key:generate
   ```

4. **تكوين قاعدة البيانات**
   قم بتحديث ملف `.env` ببيانات اتصال قاعدة البيانات:
   ```env
   DB_CONNECTION=mysql
   DB_HOST=127.0.0.1
   DB_PORT=3306
   DB_DATABASE=store_management
   DB_USERNAME=your_username
   DB_PASSWORD=your_password
   ```

5. **تشغيل الترحيلات**
   ```bash
   php artisan migrate
   php artisan db:seed
   ```

6. **بناء الأصول**
   ```bash
   npm run build
   ```

7. **مسح التخزين المؤقت**
   ```bash
   php artisan optimize:clear
   ```

8. **تشغيل الخادم**
   ```bash
   php artisan serve
   ```

### التشغيل في بيئة الإنتاج

```bash
# تحسين التطبيق
php artisan optimize

# تكوين التخزين المؤقت
php artisan config:cache
php artisan route:cache

# تشغيل عمال الطابور
php artisan queue:work --daemon
```
